---
title: Schema Boundaries
description: How SMTA segments functionality across PostgreSQL schemas.
---

SMTA uses five PostgreSQL schemas to enforce security boundaries and separate concerns. Direct table access from the `public` schema is removed ‚Äî all activity is routed through `public.*` functions.

## `core` Schema

The heart of SMTA. Contains:

- **`core.organizations`** ‚Äî top-level tenants
- **`core.organizations_meta`** ‚Äî extended metadata for organizations
- **`core.units`** ‚Äî sub-groups within organizations
- **`core.unit_meta`** ‚Äî extended metadata for units
- **`core.memberships`** ‚Äî links users to organizations with roles
- **`core.unit_memberships`** ‚Äî links users to units with roles
- **`core.roles`** ‚Äî named roles scoped per organization
- **`core.invitations`** ‚Äî pending invitations to join an organization
- **`core.audit_logs`** ‚Äî immutable record of data changes
- **`core.organization_files`** ‚Äî file metadata scoped to organizations
- **`core.users_meta`** ‚Äî extended profile data for authenticated users

RLS is enabled on all `core` tables. Policies use `core.get_current_user_id()` to determine the authenticated user.

## `platform` Schema

Service-role-only tables for SaaS-wide management. Application code cannot read or write `platform` tables ‚Äî only database service roles (your backend, not your end users) can access them.

Contains:
- **`platform.billing_customers`** ‚Äî links organizations to payment processor customer records
- **`platform.billing_subscriptions`** ‚Äî active subscription records per organization
- **`platform.subscription_products`** ‚Äî billable products synced from the payment processor
- **`platform.platform_subscription_overrides`** ‚Äî admin-applied subscription overrides
- **`platform.platform_organizations`** ‚Äî SaaS-wide organization registry for admin use
- **`platform.platform_users`** ‚Äî SaaS-wide user registry for admin use
- **`platform.platform_roles`** ‚Äî global role definitions
- **`platform.platform_feature_flags`** ‚Äî org-level feature flag overrides
- **`platform.platform_settings`** ‚Äî global SaaS configuration settings
- **`platform.platform_action_logs`** ‚Äî admin action audit trail
- **`platform.platform_system_events`** ‚Äî system-level event log
- **`platform.tenant_secrets`** ‚Äî per-tenant secret references (vault IDs)

## `utils` Schema

Shared utility functions used across all schemas. Contains helpers for soft deletion, timestamp management, and audit log insertion. Not called directly by client code.

## `public` Schema

The RPC interface. Contains only functions ‚Äî no tables. These are the only entry points for client code. In Supabase, PostgREST exposes them automatically as REST endpoints. In Payload, you call them via your database client.

See the [Public RPC Reference](/rpc-reference/organizations/) for all available functions.

## `app` Schema

Created by SMTA but not populated ‚Äî this is your domain. You define your application tables here and write RLS policies that reference `core` tables and call `core.get_current_user_id()`. SMTA's RLS enforcement cascades into your app schema through the session context.
